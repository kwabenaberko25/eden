# Skill: EdenDoc (The Educational Sage)

EdenDoc is the primary engine for creating and maintaining the "Master Class" level documentation for the Eden framework. It shifts from simple reference documentation to a **Pedagogical (Teaching) Model**, ensuring any developer—regardless of experience—can build complex applications with Eden by following verified patterns.

## 🌿 The EdenDoc "No-Assumptions" Standard

Every piece of documentation, tutorial, or recipe must adhere to these unified standards:

1. **Level-Based Granularity**:
   - **Foundational (The "Duh" Examples)**: Document the "obvious." Never assume the user knows how to import a specific internal class or set an environment variable.
   - **Integration (The "Connective Tissue")**: Every guide MUST show how the featured component connects to at least one other part of the framework (e.g., Template -> ORM, ORM -> Signal, Signal -> WebSocket).
   - **Scalability (The "Production" Path)**: Show the "Intermediate" and "Advanced" versions of the same feature to demonstrate performance and maintainability.

2. **Tutorial & Recipe Structures**:
   - **Tutorials (Option A)**: End-to-end paths (e.g., "From Data Source to Live Dashboard"). Focus on the **Mental Model** and architectural flow.
   - **Recipes (Option B)**: Problem-Solution pairs. "How do I update a UI element when a database record changes?" -> Direct pattern using HTMX and Signals.

3. **Visual & Aesthetic Excellence**:
   - **Admonitions**: Use GitHub-flavored alerts (`[!TIP]`, `[!NOTE]`, `[!IMPORTANT]`, `[!WARNING]`, `[!CAUTION]`) strategically to highlight critical framework nuances.
   - **Visual Aids**: Every Tutorial must include a **Mermaid Sequence Diagram** tracing the lifecycle of the data/request through the framework layers.
   - **Hierarchy**: Clean `<h1>` through `<h3>` structure with descriptive, accessible heading names.

## ✅ Auto-Verification Protocol

All code snippets and blocks generated by this skill MUST be verified before finalization:

1. **Extract & Parse**: Use `scripts/snippet_validator.py` to extract all Python blocks from the draft.
2. **Lint & Run**: Snippets are linted for imports and executed against the framework to ensure zero regressions.
3. **Requirement**: Any guide with unverified snippets must be marked as `[!WARNING] UNVERIFIED CODE` until fixed.

## 🛠️ Extraction & Synthesis Methodology

To ensure 100% accuracy, always source information **solely from the codebase**:

1. **Discovery Phase**: Use `grep_search` to find every instance of a feature's usage in `eden/` and `tests/`.
2. **Extraction Phase**: Analyze:
   - **Docstrings**: The primary source for "what" and "how."
   - **Type Hints**: To document parameter types and return values correctly.
   - **Context Managers/Decorators**: To understand lifecycle and injection mechanics.
3. **Synthesis Phase**: Combine code facts from research with the "Premium" pedagogical templates.

## 🛡️ Consistency & Quality Guard

- **Content Preservation**: NEVER remove existing working examples, detailed explanations, or established patterns without a critical technical reason.
- **Expansion vs. Replacement**: Always favor merging and expanding existing documentation over complete overwrites.
- **Internal Cross-Linking**: Ensure new guides link back to related concepts (e.g., *ORM* linking to *Multi-Tenancy*).
- **No Placeholders**: Never use placeholder text or "TODO" in documentation.

## 🎓 Specialized Knowledge Areas

- **The Data Forge**: Documenting `QuerySet` expansions, `annotate`, `aggregate`, and complex `Q` expressions.
- **Directive Mastery**: Maintaining guides for `@if`, `@for`, `@auth`, and `@reactive` with correct **brace-style syntax**.
- **Real-Time Synergy**: Explaining the integration between `@reactive`, `ConnectionManager`, and Background Tasks.
- **Tenancy Safety**: Proactively documenting multi-tenant isolation patterns for all database operations.
