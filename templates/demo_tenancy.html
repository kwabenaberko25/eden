How to set tenant:

💡 In production, tenants are resolved from:

  • Subdomain: acme.example.com
  • Header: X-Tenant-ID
  • Session: _tenant_id
  • Path: /t/acme/...

Features

  • ✅

    TenantMiddleware

    4 resolution strategies: subdomain, header, session, path

  • ✅

    TenantMixin

    Automatic query filtering by tenant_id

  • ✅

    Multi-Schema Support

    Each tenant gets dedicated PostgreSQL schema

  • ✅

    Fail-Secure Enforcement

    Reject requests without valid tenant (default)

  • ✅

    ContextVar Isolation

    Async-safe tenant context per request

Code Example

from eden import Eden
from eden.tenancy import TenantMixin
from eden.db import Model

# Enable tenancy
app.add_middleware("tenant", strategy="header", header_name="X-Tenant-ID")

# Models with automatic isolation
class Project(Model, TenantMixin):
    name: str
    owner_id: str
    # tenant_id automatically added and filtered

@app.get("/projects")
async def list_projects(request: Request):
    # Queries auto-filtered by current tenant
    projects = await Project.all()
    return {"projects": projects}

@app.post("/projects")
async def create_project(request: Request, data: dict):
    tenant = request.tenant  # Auto-resolved
    project = await Project.create(
        name=data["name"],
        tenant_id=tenant.id
    )
    return {"id": project.id}

🔐 Isolation

Data is automatically filtered by tenant. Queries never leak across tenants.

📊 Schemas

Each tenant can have dedicated database schema (PostgreSQL)

⚡ Performance

Indexed tenant_id columns + connection pooling per schema

Testing Multi-Tenant

This demo is configured to use the "header" strategy. To test:

  1. Try without header: Should see "Public/No Tenant"
  2. Try with different X-Tenant-ID values: Different tenant contexts
  3. Test with curl: curl -H "X-Tenant-ID: acme" http://localhost:8001/api/tenant-info
